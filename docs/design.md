# RideCircle Carpool Marketplace -- Design Document

## 1. Business Goal

Provide a fully offline, private-community carpool coordination platform where passengers post trip requests and drivers accept them, with complete lifecycle management, ratings, content moderation, and data governance -- all running on local infrastructure without any internet dependency.

## 2. User Roles

| Role          | Description |
|---------------|-------------|
| Regular User  | Can register, log in, post trip requests (as passenger), browse and accept requests (as driver), leave reviews, manage own profile. A single user can act as both passenger and driver. |
| Moderator     | All Regular User capabilities plus: review moderation queue, approve/reject flagged content, view moderation dashboards. |
| Administrator | All Moderator capabilities plus: manage organization settings, view governance dashboards, audit logs, resolve disputes, manage roles/permissions, configure system parameters. |

All roles are scoped to a single organization. A user belongs to exactly one organization.

## 3. Organization Boundary

- Each deployment serves one organization (or a small set of organizations sharing a database with `organization_id` tenant isolation).
- All data (users, listings, orders, reviews, media, events) is partitioned by `organization_id`.
- Cross-organization data access is not permitted at the application layer.
- Organization settings (name, allowed domains for hotlink protection, moderation thresholds) are configurable by Administrators.

## 4. Architecture

```
+-------------------+          JSON/REST          +-------------------+
|                   |  <========================> |                   |
|  Layui Frontend   |        HTTP over LAN        |  ThinkPHP Backend |
|  (Nginx static)   |                             |  (PHP-FPM)        |
+-------------------+                             +--------+----------+
                                                           |
                                                  +--------v----------+
                                                  |    MySQL 8.0      |
                                                  |  (local server)   |
                                                  +-------------------+
                                                           |
                                                  +--------v----------+
                                                  |  Local Disk       |
                                                  |  (media storage)  |
                                                  +-------------------+
```

- **Frontend**: Static HTML/JS/CSS served by Nginx. All Layui assets bundled locally. No CDN, no remote fonts.
- **Backend**: ThinkPHP application running behind PHP-FPM. Serves REST JSON API only -- no server-rendered HTML.
- **Database**: MySQL 8.0 with utf8mb4 encoding.
- **Media**: Local filesystem. Access via signed URLs generated by the backend.

## 5. Frontend/Backend Separation

- The frontend is a fully static single-page-like application. It makes AJAX requests to the backend API.
- The backend never serves HTML views to end users. It only returns JSON responses.
- CORS is configured to allow the frontend origin (same machine or LAN IP).
- The frontend handles all routing, rendering, form validation (client-side), and UI state.
- The backend handles all business logic, server-side validation, persistence, and authorization.

## 6. API Contract Shape

All API responses follow a consistent envelope:

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

- `code`: 0 for success, non-zero for errors (mapped to business error codes).
- `message`: Human-readable message (English).
- `data`: Payload object (null on error unless partial data is relevant).

Error responses additionally include:

```json
{
  "code": 40001,
  "message": "Validation failed",
  "data": null,
  "errors": {
    "rider_count": ["Rider count must be between 1 and 6"]
  }
}
```

HTTP status codes are used correctly (200, 201, 400, 401, 403, 404, 409, 422, 429, 500).

## 7. Auth Model

### 7.1 Session Login
- Users authenticate via `POST /api/auth/login` with email/password.
- Backend creates a server-side session and returns a session cookie.
- Session lifetime is configurable (default: 120 minutes).
- Sessions are stored in the database (not filesystem) for reliability.

### 7.2 API Token
- Users can generate long-lived API tokens via `POST /api/auth/tokens`.
- Tokens are sent in the `Authorization: Bearer <token>` header.
- Default expiry: 90 days.
- Token rotation: users can rotate a token (new token issued, old one invalidated) via `POST /api/auth/tokens/{id}/rotate`.
- Tokens are stored as SHA-256 hashes in the database. The plaintext is shown only once at creation.
- Each user can have multiple active tokens (e.g., for different integrations).

A user can have multiple roles. Effective permissions are the union of all assigned role permissions.

## 10. Object-Level Authorization Model

Beyond RBAC, resource-level ownership checks are enforced:

- **Listings**: Only the creator can edit/delete their own listing. Moderators can moderate any listing. Administrators can manage all.
- **Orders**: Only the passenger (listing creator) and the matched driver can view/act on an order. Administrators can view all orders.
- **Reviews**: Only the author can edit/delete their own review (before moderation). Moderators can hide/remove any review.
- **Media**: Media inherits access control from its parent resource (review, listing). Signed URLs are scoped to the requesting user's permissions.
- **Moderation Queue**: Only Moderators and Administrators can access.
- **Governance Dashboards**: Only Administrators can access.
- **Audit Logs**: Only Administrators with explicit `audit.read` permission can access. Masked by default; unmasked view requires `audit.read_unmasked` permission.

Implementation: ThinkPHP middleware checks RBAC first, then a policy class checks object ownership.

## 11. Listing Workflow

### States
```
[draft] --> [active] --> [matched] --> [in_progress] --> [completed]
   |           |            |              |                  |
   |           v            v              v                  v
   |       [canceled]   [canceled]     [canceled*]       [disputed]
   |                                                         |
   v                                                         v
[deleted]                                               [resolved]
```

*Cancellation from in_progress is blocked at the order level; listing stays in_progress.

### Transitions
| From        | To          | Trigger | Conditions |
|-------------|-------------|---------|------------|
| draft       | active      | Publish | All required fields valid |
| draft       | deleted     | Delete  | Owner only |
| active      | draft       | Unpublish | Owner or Moderator |
| active      | matched     | Driver accepts | Order created |
| active      | canceled    | Owner cancels or bulk close | No active order |
| matched     | in_progress | Either party starts trip | Order state = accepted |
| matched     | canceled    | Cancellation rules apply | Via order cancellation |
| in_progress | completed   | Trip completed | Both parties or driver marks complete |
| completed   | disputed    | Dispute opened | Within 72-hour window |
| disputed    | resolved    | Admin resolves | Administrator action |

### Bulk Close
- Administrators can select multiple active listings and close them as "canceled" with reason "outdated".
- Only active listings with no matched order can be bulk closed.

- Re-publishing after edit shows the diff to the user for confirmation.

## 13. Order Lifecycle

### States
```
[pending_match] --> [accepted] --> [in_progress] --> [completed]
       |                |               |                 |
       v                v               |                 v
  [expired]        [canceled]           |            [disputed]
                                        |                 |
                                   (blocked)              v
                                                     [resolved]
```

### State Machine Rules

| Transition            | Conditions |
|-----------------------|------------|
| pending_match -> expired | 30 minutes elapsed with no driver acceptance (auto, via scheduled task) |
| pending_match -> accepted | Driver accepts the listing |
| accepted -> in_progress | Either party marks trip as started |
| accepted -> canceled | Within 5 minutes of acceptance: free cancel (no reason required). After 5 minutes: reason code required. |
| in_progress -> completed | Driver or both parties mark trip complete |
| in_progress -> canceled | **BLOCKED** -- cancellation not allowed once in progress |
| completed -> disputed | Within 72 hours of completion timestamp |
| disputed -> resolved | Administrator resolves with resolution notes |

### Auto-Expiry Implementation
- A scheduled command (`php think order:expire`) runs every minute via cron.
- Finds all orders with `status = pending_match` and `created_at < NOW() - 30 minutes`.
- Transitions them to `expired` and restores the listing to `active`.

### Cancellation Reason Codes
| Code | Description |
|------|-------------|
| DRIVER_UNAVAILABLE | Driver can no longer make the trip |
| PASSENGER_CHANGED_PLANS | Passenger no longer needs the ride |
| VEHICLE_ISSUE | Vehicle problem prevents the trip |
| SCHEDULE_CONFLICT | Time conflict arose |
| OTHER | Free-text reason required |

## 14. Moderation Engine

### Components

1. **Sensitive-Word Filter**
   - A local text file contains the word list (shipped with deployment, editable by Admins).
   - Applied to: review text, listing titles, listing descriptions, user display names.
   - Matching: case-insensitive substring match with word-boundary awareness.
   - Action: flag for manual review; content is not published until approved.

2. **Rate Limiting**
   - Max 3 reviews per user per hour (sliding window).
   - Enforced at API layer. Returns HTTP 429 with retry-after header.

3. **Duplicate Detection**
   - **Text**: N-gram similarity (trigram Jaccard index) between new review text and recent reviews by the same user. Threshold configurable (default: 0.85).
   - **Files**: SHA-256 fingerprint of uploaded files. Exact-match detection against all files uploaded by the same user in the last 30 days.
   - Action: flag for manual review if duplicate detected.

4. **Credibility Score**
   - Computed per-review to weight its influence on aggregate ratings.
   - Formula: `score = (W_age * age_factor) + (W_completion * completion_factor) + (W_pattern * pattern_factor)`
   - **age_factor**: 0.5 if account < 14 days old, 1.0 otherwise.
   - **completion_factor**: `completed_orders / total_orders` (0 if no orders).
   - **pattern_factor**: 1.0 baseline, reduced by 0.25 for each anomaly (e.g., all-5-star reviews in burst, review immediately after order without plausible trip time).
   - Weights configurable via `.env` (default: 0.3, 0.4, 0.3).
   - Score range: 0.0 to 1.0. Reviews with score < 0.3 are auto-flagged.

5. **Manual Moderation Queue**
   - Flagged items appear in a queue visible to Moderators and Administrators.
   - Actions: approve (publish), reject (hide with reason), escalate to Administrator.
   - Queue shows: item type, content preview, flag reason, credibility score, user history summary.

## 15. Duplicate Detection Approach

Two-layer strategy, both fully local:

### Text Similarity
- Extract character trigrams from the review text.
- Compute Jaccard similarity coefficient against recent reviews (same user, last 30 days).
- If similarity >= threshold (default 0.85), flag as potential duplicate.
- Implementation: PHP function, no external library required.

### File Fingerprint
- On upload, compute SHA-256 hash of file contents.
- Store hash in `media.file_hash` column.
- Before accepting a new upload, check for existing files with the same hash (same user, last 30 days).
- If match found, flag and optionally reject with "duplicate file" message.

## 16. Credibility Score Model

See Section 14.4 for the formula. Additional details:

### Storage
- Computed at review-write time and stored in `reviews.credibility_score`.
- Recomputed nightly for all reviews during governance jobs (accounts age, completion stats change).

### Usage
- Weighted average rating: `SUM(rating * credibility_score) / SUM(credibility_score)` per listing/driver.
- Displayed to users as the listing/driver rating (not the raw average).
- Administrators can view raw vs. weighted ratings on governance dashboards.

### Anomaly Patterns
| Pattern | Penalty |
|---------|---------|
| 3+ five-star reviews from same user within 1 hour | -0.25 per review in burst |
| Review posted within 5 minutes of order completion | -0.25 (implausible trip time) |
| Identical rating and near-identical text across multiple reviews | -0.25 |

## 17. Media Storage and Signed URL Model

### Storage
- All media files are stored on local disk under `MEDIA_STORAGE_PATH`.
- Directory structure: `/{org_id}/{year}/{month}/{day}/{uuid}.{ext}`
- Original filename is stored in the database but not used for disk storage (prevents path traversal).

### Database: `media`
| Column       | Type         | Description |
|--------------|--------------|-------------|
| id           | BIGINT PK    | Auto-increment |
| org_id       | BIGINT FK    | Organization |
| user_id      | BIGINT FK    | Uploader |
| parent_type  | VARCHAR(30)  | Polymorphic: review, listing |
| parent_id    | BIGINT       | Parent record ID |
| file_name    | VARCHAR(255) | Original filename |
| file_path    | VARCHAR(512) | Disk path relative to storage root |
| file_hash    | VARCHAR(64)  | SHA-256 hash |
| file_size    | BIGINT       | Size in bytes |
| mime_type    | VARCHAR(100) | MIME type |
| file_type    | ENUM         | photo, video |
| watermarked  | BOOLEAN      | Whether watermark has been applied |
| created_at   | DATETIME     | Upload timestamp |

### Signed URL Generation
- URL format: `/api/media/{id}?signature={sig}&expires={ts}`
- Signature: `HMAC-SHA256(media_id + expires_timestamp, MEDIA_SIGN_SECRET)`
- Default expiry: 10 minutes from generation.
- Backend validates signature and expiry on every media request.

### Upload Validation
| File Type | Max Size | Allowed MIME Types |
|-----------|----------|--------------------|
| Photo     | 5 MB     | image/jpeg, image/png, image/gif, image/webp |
| Video     | 50 MB    | video/mp4, video/webm |
| Per review | Max 5 files total | Mixed photo+video allowed |

## 18. Watermark Model

- **When**: Applied on upload if `MEDIA_WATERMARK_ENABLED=true`.
- **What**: A semi-transparent PNG overlay positioned at the bottom-right corner of images.
- **How**: PHP GD library (`imagecopy` with alpha blending) or Imagick if available.
- **Scope**: Photos only (not videos).
- **Storage**: The watermarked version replaces the original on disk. The original is not retained (to save disk space in offline deployments).
- **Configurable**: Watermark image path, position, and opacity via `.env`.

## 19. Hotlink Protection Model

- **Signed URL Validation**: Every media request must include a valid, non-expired signature (see Section 17).
- **Referrer Check**: The `Referer` header must match one of the allowed domains in `HOTLINK_ALLOWED_DOMAINS`.
- **Enforcement**: ThinkPHP middleware checks both conditions. Requests failing either check receive HTTP 403.
- **Fallback**: If the `Referer` header is missing (e.g., direct browser navigation), the signed URL alone is sufficient -- but the URL's short expiry (10 minutes) limits abuse.

## 20. Data Governance Model

### Event Capture
All user interactions generate behavioral events stored in `behavior_events`:

| Column       | Type         | Description |
|--------------|--------------|-------------|
| id           | BIGINT PK    | Auto-increment |
| org_id       | BIGINT FK    | Organization |
| user_id      | BIGINT FK    | Acting user (nullable for anonymous browse) |
| event_type   | VARCHAR(30)  | browse, click, favorite, rate, search, listing_view |
| target_type  | VARCHAR(30)  | listing, order, review, user |
| target_id    | BIGINT       | Target record ID |
| metadata     | JSON         | Additional event data (search query, filter params, etc.) |
| created_at   | DATETIME     | Event timestamp |

Events are written asynchronously (queued in-memory, flushed in batch) to minimize request latency impact.

### Listing Metadata
Listing records include governance-relevant fields:
- `view_count`, `favorite_count`, `click_count` (denormalized counters, updated from events).
- `last_activity_at` (latest event timestamp for this listing).
- `data_quality_score` (computed nightly, see Section 23).

## 21. Nightly Jobs Model

Scheduled via cron, implemented as ThinkPHP console commands:

| Command | Schedule | Description |
|---------|----------|-------------|
| `php think governance:dedup` | 2:00 AM | Deduplicate behavior events (identical user+type+target within 1-minute window) |
| `php think governance:fill` | 2:15 AM | Fill missing values in listing metadata (e.g., compute counters from events if denormalized counters drifted) |
| `php think governance:lineage` | 2:30 AM | Record lineage entries for each cleanup step executed |
| `php think governance:quality` | 2:45 AM | Compute data quality metrics for Administrator dashboard |
| `php think credibility:recompute` | 3:00 AM | Recompute credibility scores for all reviews |
| `php think order:expire` | Every minute | Auto-expire pending_match orders past 30 minutes |

### PII Masking
- By default, `user_id` and `ip_address` are masked in log views (e.g., `user_***42`, `192.168.***.***`).
- Administrators with `audit.read_unmasked` permission see full values.
- Password hashes and token secrets are **never** included in audit log values.

### Application Logs
- ThinkPHP's built-in logging writes to local files.
- Log levels: debug, info, warning, error.
- PII masking is applied at the log-write layer: email addresses, phone numbers, and IPs are masked unless the log consumer has audit authorization.
- Log rotation: daily files, retained for 90 days.

## 25. Deployment Model

### Docker Compose (Primary)
- Three containers: MySQL, Backend (PHP-FPM + Nginx sidecar or combined), Frontend (Nginx).
- All containers on a single Docker network.
- Volumes for MySQL data persistence and media storage.
- `.env` file configures all runtime parameters.

### Manual Deployment (Alternative)
- Install PHP 8.x, MySQL 8.0, Nginx on same machine.
- Configure Nginx to serve frontend static files and proxy `/api/*` to PHP-FPM.
- Set up cron jobs for nightly governance commands and per-minute order expiry.

### No Internet Required
- All PHP dependencies vendored in `backend/vendor/` (committed or packaged).
- All Layui assets in `frontend/layui/` (committed).
- No Composer or npm install required at deployment time.
- MySQL image can be pre-pulled or loaded from a local tarball.
